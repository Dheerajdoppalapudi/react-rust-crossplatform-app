Drawbacks of illustrations flow:
      LLMs are very bad at spatial reasoning in pixel coordinates. They don't "see" a canvas. They're pattern-matching from training data, not reasoning about space.
      The shape vocabulary is too primitive. Rectangles + ellipses + lines can't make a convincing robot doing a flip, let alone explain thermodynamics with a meaningful visual.
      There's no feedback loop. The LLM generates blind — it can't see the output and fix proportions.
      Thermodynamics-style concept diagrams (molecules, heat flow, pressure) are particularly bad here because they need curves, gradients, and accurate spatial metaphors that require real drawing tools.

Request lifecycle (API call example)

1. User clicks something in React UI
2. api.js calls https://d2802jabx2c8dk.cloudfront.net/api/conversations
3. CloudFront matches /api/* behavior → forwards to EC2
4. Nginx on EC2 receives request → proxy_pass to 127.0.0.1:8000
5. FastAPI handles the route → queries SQLite → returns JSON
6. Response travels back through Nginx → CloudFront → browser
Deploy lifecycle

1. You push code to GitHub main branch
2. GitHub Actions triggers deploy.yml
3. Frontend job: npm run build → aws s3 sync → cloudfront invalidation
4. Backend job: ssh into EC2 → git pull → pip install → systemctl restart
5. New code is live within ~2 minutes

Pattern: Fire and poll


POST /api/generate_video → returns immediately
  { "job_id": "abc123", "status": "queued" }

GET /api/video_status/{job_id} → UI polls every 2-3s
  { "status": "processing", "stage": "tts", "progress": 40 }
  { "status": "done", "video_url": "/outputs/..." }
Implementation options:

FastAPI BackgroundTasks — simplest, no new infrastructure, runs in same process
Celery + Redis — proper task queue, survives restarts, scales to multiple workers
Celery + SQS — cloud-native if you're on AWS
Trade-off: BackgroundTasks is simple but if the server restarts mid-job, the task is lost. Celery persists jobs.
